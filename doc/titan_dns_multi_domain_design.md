# Titan-DNS 强控轮询调度方案 (v4 重构版)

本文定义了 Titan-DNS 如何通过服务端负载均衡（Server-Side LB），将流量精确分发到单机多网卡物理口。

## 1. 解析响应策略：单记录循环 (Single-Record Round-Robin)
DNS 服务器每次响应 A 记录请求时，**仅返回一个且是最优的一个 IP 记录**，不再返回完整的 IP 列表。

### 1.1 带来的核心价值
*   **强控流量分布**：由于 DNS 决定了唯一的请求入口，客户端无需处理多 A 记录逻辑，流量平摊完全由 DNS 控制。
*   **降低 DNS 包大小**：只返回一个 IP，解析响应包极小，处理速度更快。

---

## 2. 动静分流与负载平衡逻辑

根据域名级数的不同，采取差异化的 IP 选择算法：

### 2.1 固定域名模式 (四级域名)
*   **示例**：`pop1.pop.example.com`
*   **算法**：**全局轮询 (Global Round-Robin)**
*   **逻辑**：内存中为 `pop1` 维护一个共享的 `Index`。每次有针对该域名的解析进来，`Index = (Index + 1) % IP_Count`，依次返回网卡 IP 1 至 8。
*   **应用**：适用于普通的集群入口、管理域名。

### 2.2 动态域名模式 (五级域名/Session)
*   **示例**：`{session}.pop1.pop.example.com`
*   **算法**：**一致性映射 (Sticky Hash-based RR)**
*   **逻辑**：
    1.  提取前面的 `{session}` 部分。
    2.  计算 `Target_Index = Hash(session) % IP_Count`。
    3.  **固定返回**：对于同一个 Session 名，永远返回同一个物理口 IP。
*   **应用**：专为 **Socks5 持久连接** 设计。确保同一个移动设备/IoT 设备在重连期间由于 DNS 缓存刷新，依然能回到原来的物理网口，防止由于 IP 变化导致的 Session 掉线。

---

## 3. POP 资源组动态管理 (API)

### 3.1 核心配置模型
通过 API 更新 POP 资源时，DNS 服务器实时重置该 POP 的轮询序列：

```json
{
  "pop_id": "pop1",
  "active_ips": ["14.1.1.1", "14.1.1.2", "14.1.1.3", "..."],
  "poll_ptr": 0 // 内存中的当前轮询位置
}
```

### 3.2 动态摘除
API 配合监控系统，若物理口发生掉线，调用 `/api/v1/pop/remove_ip` 实时缩减 `active_ips`。DNS 解析会在下一个请求时自动跳过故障口，实现 **无损测速/无感切换**。

---

## 4. 后端存储与更新同步
*   **配置中心**：Redis 存储各 POP 节点的物理资源。
*   **实时同步**：DNS 节点通过 `SUBSCRIBE` 订阅 Redis 频道，在收到 IP 资源变动（如 PPPoE 重拨换网）后，30ms 内完成解析记录的内存更新。
