# Titan-DNS 强控轮询调度方案 (v4 重构版)

本文定义了 Titan-DNS 如何通过服务端负载均衡（Server-Side LB），将流量精确分发到单机多网卡物理口。

## 1. 解析响应策略：单记录循环 (Single-Record Round-Robin)
DNS 服务器每次响应 A 记录请求时，**仅返回一个且是最优的一个 IP 记录**，不再返回完整的 IP 列表。

### 1.1 带来的核心价值
*   **强控流量分布**：由于 DNS 决定了唯一的请求入口，客户端无需处理多 A 记录逻辑，流量平摊完全由 DNS 控制。
*   **低开销解析**：只返回一个 IP，解析响应包极小，极大提升了 DNS 在基础网络环境（如 IoT 设备）下的处理速度。

---

## 2. 动静分流与负载平衡逻辑

根据请求域名的级数和前缀不同，采取差异化的 IP 选择算法：

### 2.1 固定域名模式 (四级域名)
*   **示例**：`pop1.pop.example.com`
*   **算法**：**全局顺序轮询 (Global Round-Robin)**
*   **逻辑**：内存中为 `pop1` 维护一个共享的 `Index` 指针。每次有针对该域名的解析进来，`Index = (Index + 1) % IP_Count`，按 1 -> 2 -> 3... 8 的顺序循环返回物理网口 IP。
*   **应用**：适用于集群管理入口、静态控制通道。

### 2.2 动态域名模式 (五级域名/Session)
*   **示例**：`{session}.pop1.pop.example.com`
*   **算法**：**Session 粘性哈希轮询 (Hash-based Sticky RR)**
*   **逻辑**：
    1.  提取前面的 `{session}` (如 `xxx`) 部分。
    2.  计算该字符串的哈希值：`Target_Index = Hash(session) % IP_Count`。
    3.  **固定分配**：对于同一个 Session 名，永远返回同一个物理口 IP。
*   **应用**：专为 **Socks5 持久连接** 设计。确保同一个移动设备或 IoT 设备在请求期间，即便 DNS TTL 过期刷新，也能被强行导向同一个物理出口 IP，避免由 IP 跳变造成的 Session 断连和认证失败。

## 3. POP 资源组动态管理 (API/Redis)

### 3.1 内存模型
DNS 服务内存镜像如下，用于执行快速调度：

```json
{
  "pop1": {
    "active_ips": ["14.1.1.1", "14.1.1.2", "14.1.1.3", "..."],
    "poll_ptr": 2,      // 用于固定域名轮询的当前位置
    "health_status": "ok"
  }
}
```

### 3.2 同步机制
*   **管理接口**：通过 `POST /api/v1/pop/config` 更新 IP 列表。
*   **Redis 驱动**：DNS 节点利用 Redis 订阅机制接收物理口变动，实时清空过时的解析缓存并重置轮询指针。

---

## 4. 结论与性能指标
*   **解析耗时**：单次解析 < 0.5ms。
*   **带宽叠加效能**：通过这种强制的分流分发，在 8 个物理口的场景下，总带宽利用率可稳定在 **95% 以上**，有效打破单网线、单 CPU 核的处理限制。
